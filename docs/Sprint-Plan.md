## 2. MVP 阶段详细拆解（第 1-4 周）

### 2.1 第 1 周：项目初始化与基础架构

#### 任务 T1.1 — 项目脚手架搭建

- **描述**：使用 Vite 创建 React + TypeScript 项目，配置 ESLint、Prettier、路径别名，安装核心依赖（PixiJS、Zustand、React Router、Dexie.js、GSAP）
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：无
- **验收标准**：
  - `pnpm dev` 启动开发服务器，HMR 正常工作
  - `pnpm build` 构建成功，无报错
  - ESLint / Prettier 配置生效，保存自动格式化
  - `@/` 路径别名可正确解析
- **技术细节**：使用 `pnpm` 作为包管理器，配置 `tsconfig.json` 的 paths，安装 `vite-plugin-svg-sprites` 用于图标管理

#### 任务 T1.2 — 项目目录结构与模块划分

- **描述**：建立标准化的项目目录结构，定义各模块的职责边界和导出接口
- **预估工时**：0.5 人天
- **优先级**：P0
- **前置依赖**：T1.1
- **验收标准**：
  - 目录结构（MVP 阶段）：`src/core/`（游戏核心）、`src/ui/`（React 组件）、`src/stores/`（Zustand stores）、`src/systems/`（各子系统）、`src/config/`（数值配置）、`src/utils/`（工具函数）、`src/assets/`（静态资源）
  - 注：MVP 采用单应用结构以降低复杂度；V1.0+ 目标迁移至 Monorepo（`packages/game-core`、`packages/game-renderer`、`packages/game-ui`、`app`），详见 `TDD.md` 第1章
  - 每个目录下有 `index.ts` 统一导出
  - `README.md` 中记录目录说明
- **技术细节**：
  ```
  src/
  ├── core/          # 游戏引擎层
  │   ├── Game.ts         # 游戏主循环
  │   ├── Scene.ts        # 场景管理
  │   └── AssetLoader.ts  # 资源加载器
  ├── systems/       # 游戏系统
  │   ├── coffee/         # 制作系统
  │   ├── customer/       # 顾客系统
  │   ├── economy/        # 经济系统
  │   ├── decoration/     # 装修系统
  │   ├── employee/       # 员工系统
  │   ├── inventory/      # 库存系统
  │   └── quest/          # 任务系统
  ├── stores/        # Zustand 状态管理
  ├── ui/            # React UI 层
  ├── config/        # 数值配置表
  └── utils/         # 通用工具
  ```

#### 任务 T1.3 — PixiJS 渲染引擎集成

- **描述**：在 React 应用中集成 PixiJS Canvas，实现基础的游戏渲染管线（初始化、resize、帧循环）
- **预估工时**：1.5 人天
- **优先级**：P0
- **前置依赖**：T1.1
- **验收标准**：
  - PixiJS Application 在 React 组件中正确挂载
  - 窗口 resize 时 Canvas 自适应缩放
  - 60fps 稳定帧率，无内存泄漏
  - React 组件与 PixiJS 的通信桥接可用（通过 EventEmitter 或 ref）
- **技术细节**：创建一个 `GameCanvas` React 组件封装 PixiJS Application，使用 `useRef` 持有 PixiJS 实例，`useEffect` 管理生命周期。导出一个全局 EventBus 用于 React 与 PixiJS 双向通信

#### 任务 T1.4 — 状态管理架构设计（Zustand）

- **描述**：设计并实现 Zustand Store 架构，定义核心游戏状态的 Shape 和各子系统 Store 的划分
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：T1.1、T1.2
- **验收标准**：
  - 完成以下 Store 的定义和基础实现：
    - `useGameStore`：游戏整体状态（阶段、时间、暂停状态）
    - `usePlayerStore`：玩家数据（金币、钻石、等级、经验）
    - `useShopStore`：店铺状态（桌椅、设备、解锁状态）
    - `useRecipeStore`：食谱数据（已解锁、品质记录）
  - 每个 Store 包含完整的 TypeScript 类型定义
  - 提供 `persist` 中间件接口（为后续 IndexedDB 持久化预留）
- **技术细节**：使用 Zustand 的 `immer` 中间件简化嵌套状态更新，使用 `devtools` 中间件支持 Redux DevTools 调试

#### 任务 T1.5 — IndexedDB 持久化层

- **描述**：基于 Dexie.js 封装数据持久化层，实现游戏存档的保存/读取/删除功能
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：T1.1、T1.4
- **验收标准**：
  - 定义存档 Schema（存档 ID、时间戳、游戏状态 Blob、版本号）
  - 实现 `saveGame()`、`loadGame()`、`deleteSave()`、`listSaves()` 四个基础 API
  - 自动存档（每 5 分钟自动保存一次，每日打烊时强制存档）
  - 手动存档（最多保留 3 个存档槽位，slotId 0-2）
  - 版本号校验（非兼容版本提示更新）
- **技术细节**：使用 Dexie.js 的 `Table` API，将 Zustand 的完整 state JSON 序列化后存储。归档时带 `version` 字段，加载时校验

#### 任务 T1.6 — UI 框架与导航系统

- **描述**：搭建 React UI 框架，实现顶部状态栏、底部导航栏、页面路由
- **预估工时**：1.5 人天
- **优先级**：P0
- **前置依赖**：T1.1、T1.2
- **验收标准**：
  - 顶部状态栏显示：游戏时间、天气图标、金币数、钻石数、店铺等级
  - 底部导航栏包含 5 个 Tab：商城、装修、员工、食谱、库存（MVP 阶段前 3 个可点击）
  - 点击 Tab 切换对应面板，面板以抽屉/模态框形式从底部/侧边弹出
  - 路由状态可通过 URL hash 同步（可选）
- **技术细节**：使用 CSS Modules 或 Tailwind CSS 做样式管理。底部导航栏使用 `fixed bottom-0` 定位，面板使用 `transform` 动画实现滑入/滑出

#### 任务 T1.7 — 资源加载器与占位素材

- **描述**：实现异步资源加载管理器，准备 MVP 阶段的占位素材（简单几何图形替代精灵图）
- **预估工时**：0.5 人天
- **优先级**：P1
- **前置依赖**：T1.3
- **验收标准**：
  - `AssetLoader` 类支持加载图片精灵、精灵图集（Spritesheet）、音频
  - 提供加载进度回调（用于 Loading 界面）
  - 加载失败有降级处理（显示占位色块）
  - 准备咖啡机、桌子、顾客、咖啡杯的简易占位图形（纯色矩形 + 标签文字）
- **技术细节**：基于 PixiJS 的 `Assets` API（`@pixi/assets`），配合 `Texture.from()` 处理单图加载

#### 任务 T1.8 — 游戏主循环与时间系统

- **描述**：实现游戏主循环（Game Loop），管理游戏内时间流逝（1 天 = 12 分钟现实时间）和游戏阶段切换
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：T1.3
- **验收标准**：
  - 游戏时间从早晨 6:00 开始，夜晚 22:00 打烊（时间基线以 `Game-Design.md` 第8章为准）
  - 时间流逝速度：正常 1x（1天=12分钟），支持 2x / 4x 加速
  - 游戏阶段自动切换：早晨→上午→中午→下午→傍晚→夜晚
  - 每个阶段切换时有视觉提示（色调变化）
  - 暂停功能：点击暂停按钮，游戏时间停止，但 UI 仍可交互
  - 打烊后自动进入"装修模式"（时间停止，顾客不再进入）
- **技术细节**：使用 PixiJS 的 `Ticker` 驱动游戏循环，通过 `deltaTime` 计算游戏内时间。设计一个 `TimeManager` 单例管理时间状态

### 2.2 第 2 周：核心制作循环与顾客系统

#### 任务 T2.1 — 咖啡厅主场景渲染

- **描述**：使用 PixiJS 渲染咖啡厅主场景（等距/平铺视角），包含地板、墙壁、基础家具的静态布局
- **预估工时**：2 人天
- **优先级**：P0
- **前置依赖**：T1.3、T1.7、T1.8
- **验收标准**：
  - 场景中可见：地板（网格纹理）、4 面墙壁（不同颜色区分）、吧台区域、2 张桌子、1 台咖啡机
  - 场景使用正交投影模拟 2.5D 等距视角（或简单的俯视 2D 平铺视角作为 MVP 降级方案）
  - 场景大小适配窗口，支持缩放和平移（至少支持缩放）
  - 各元素有正确的前后遮挡关系（通过 y 坐标 z-index 排序）
- **技术细节**：MVP 阶段建议使用简单的 2D 俯视平铺视角以降低复杂度。使用 PixiJS 的 `Container` 组织场景层级，通过 `sortableChildren` 和自定义 `sortFunction` 实现 z-sorting

#### 任务 T2.2 — 咖啡制作流程（基础 3 款）

- **描述**：实现美式、拿铁、卡布奇诺三款基础咖啡的制作流程（点击交互式步骤）
- **预估工时**：2.5 人天
- **优先级**：P0
- **前置依赖**：T2.1
- **验收标准**：
  - 点击咖啡机进入制作面板，显示当前可制作的咖啡列表
  - 选择美式制作流程：研磨咖啡豆 → 萃取浓缩 → 加热水（3 步，每步 2-3 秒等待）
  - 选择拿铁制作流程：研磨咖啡豆 → 萃取浓缩 → 打奶泡 → 倒入牛奶（4 步）
  - 选择卡布奇诺制作流程：研磨咖啡豆 → 萃取浓缩 → 打厚奶泡 → 倒入牛奶 + 奶泡（4 步）
  - 每步骤显示进度条动画和视觉反馈（机器震动/冒热气）
  - 制作完成后，成品显示在出餐区，图标+名称清晰可辨
  - 支持同时进行多个制作任务（最多 2 个，仅限有 2 个制作槽位）
- **技术细节**：设计一个 `CookingProcess` 类管理制作状态机：`IDLE → PREPARING → BREWING → STEAMING → COMPLETE`。每一步的等待时间通过 `Ticker` 计时

#### 任务 T2.3 — 顾客生成系统（Spawner）

- **描述**：实现顾客生成逻辑，根据时间段、店铺等级、当前客流量动态生成顾客
- **预估工时**：1.5 人天
- **优先级**：P0
- **前置依赖**：T1.4、T2.1
- **验收标准**：
  - 顾客按时间规律生成：早餐高峰（7:00-9:00）每分钟 2-3 位，平峰（10:00-14:00）每分钟 1 位，下午茶高峰（14:00-16:00）每分钟 2 位，晚间（17:00-21:00）每分钟 1-2 位
  - 随机从已解锁的顾客类型中选取（MVP 阶段至少包含上班族、学生、退休老人 3 种）
  - 店铺客满（所有座位被占）时暂停生成新顾客
  - 顾客数量有上限（MVP：不超过 8 人同时在店）
- **技术细节**：顾客生成器使用加权随机算法，不同时段不同类型的生成权重不同。配置表定义每种顾客的出现条件、权重、基础属性

#### 任务 T2.4 — 顾客 AI 行为状态机

- **描述**：实现顾客的完整行为状态机，包括进店、排队、点单、等待、用餐、结账、离开
- **预估工时**：2.5 人天
- **优先级**：P0
- **前置依赖**：T2.3
- **验收标准**：
  - 顾客状态机包含 7 个状态：`ENTERING → QUEUING → ORDERING → WAITING → EATING → PAYING → LEAVING`
  - ENTERING：从门口走进咖啡厅，寻找空位（1-2 秒动画）
  - QUEUING：若无空位则在门口排队等候
  - ORDERING：坐下后头顶显示气泡，随机从已解锁食谱中选择 1-2 个商品
  - WAITING：显示等待倒计时进度条（基于顾客类型耐心值），超时则生气离开
  - EATING：收到商品后开始用餐，持续 5-15 秒（取决于商品类型）

### 3.1 第 5 周：员工系统

#### 任务 T5.1 — 员工数据模型与配置

- **描述**：设计员工系统的数据模型，定义 5 种职位的属性、技能、升级曲线
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：T1.4
- **验收标准**：
  - 员工数据模型：
    ```
    interface Employee {
      id: string;
      name: string;
      role: 'barista' | 'pastry_chef' | 'waiter' | 'cleaner' | 'musician';
      level: number;          // 1-10
      exp: number;            // 当前经验值
      mood: number;           // 0-100 心情值
      salary: number;         // 基础薪资（随等级递增）
      skills: Skill[];        // 已解锁技能
      isWorking: boolean;     // 是否当班
      hireDate: number;       // 雇佣时间戳
    }
    ```
  - 5 种职位配置，定义各自的属性加成：
    - 咖啡师：每级 +8% 制作速度, +5% 品质, 每3级 +1 同时制作数
    - 甜品师：每级 +8% 制作速度, +5% 品质
    - 服务员：每级 +10% 移动速度, 每2级 +1 端盘数
    - 清洁工：每级 +12% 清洁速度, 提升整洁度上限
    - 驻唱歌手：每级 +5% 顾客耐心延长, +2% 小费概率
  - 升级经验曲线：`expToNextLevel = level * 100`（1→2 需 100exp, 9→10 需 900exp）
  - 心情值机制：工作每小时 -5 心情，发工资 +30 心情，发奖金 +50 心情，休息 +20 心情/小时
  - 所有配置定义在 `config/employees.ts`
- **技术细节**：员工配置表使用 TypeScript `as const` 确保类型安全，导出 `EMPLOYEE_ROLES` 常量

### 3.3 第 7 周：任务成就与随机事件

#### 任务 T7.1 — 任务系统架构

- **描述**：设计并实现可扩展的任务系统架构，支持每日任务、成就、随机事件三类任务
- **预估工时**：1.5 人天
- **优先级**：P0
- **前置依赖**：T1.4
- **验收标准**：
  - 任务基类：
    ```
    interface Quest {
      id: string;
      type: 'daily' | 'achievement' | 'event';
      title: string;
      description: string;
      objective: QuestObjective;   // 任务目标
      progress: number;             // 当前进度
      target: number;               // 目标值
      rewards: Reward[];            // 奖励列表
      status: 'locked' | 'active' | 'completed' | 'claimed';
    }
    ```
  - `QuestObjective` 支持多种类型：
    - `make_drink`：制作 N 杯饮品
    - `serve_customer`：服务 N 位顾客
    - `earn_gold`：累计赚取 N 金币
    - `get_good_review`：获得 N 次好评
    - `upgrade_equipment`：升级 N 次设备
    - `place_furniture`：放置 N 件家具
    - `hire_employee`：雇佣 N 名员工
    - `composite`：组合条件（AND/OR）
  - 进度追踪系统（`QuestTracker`）监听游戏事件，自动更新相关任务进度
  - 任务完成时触发回调，弹出完成通知
- **技术细节**：使用观察者模式，各游戏系统在关键节点 emit 事件，`QuestTracker` 订阅事件并更新对应任务进度

#### 任务 T7.2 — 每日任务系统

- **描述**：实现每日随机刷新的日常任务，包含 3 个每日目标和奖励
- **预估工时**：1 人天
- **优先级**：P0
- **前置依赖**：T7.1
- **验收标准**：
  - 每天（游戏内天数切换时）自动刷新 3 个每日任务
  - 每日任务示例：
    - "热卖日"：今日卖出 20 杯咖啡 → 奖励 500G
    - "好评如潮"：今日获得 10 次好评 → 奖励 1💎
    - "新品推荐"：今日制作 3 种不同类型的饮品 → 奖励 300G + 50EXP
  - 每日任务显示在主界面右侧或顶部可展开区域
  - 每个任务显示进度条（如 15/20 杯）
  - 完成的任务显示打勾标记，"领取奖励"按钮（点击获得奖励）
  - 每日任务重置时间 = 游戏内打烊/新一天开始
  - 未完成的每日任务不累积到第二天
  - 连续完成所有 3 个每日任务获得额外奖励（钻石 ×1）
- **技术细节**：每日任务模板池定义在 `config/quests.ts`，每天随机从池中抽取。`DailyQuestManager` 在游戏天切换时重新 roll

#### 任务 T7.3 — 成就系统

- **描述**：实现成就系统，包含多层级成就和里程碑奖励
- **预估工时**：1.5 人天
- **优先级**：P1
- **前置依赖**：T7.1
- **验收标准**：
  - 成就分为 5 大类（参照 PRD）：经营成就（10个）、制作成就（8个）、顾客成就（6个）、装修成就（6个）、员工成就（4个）、隐藏成就（3个）
  - 成就通常有多个层级（如"累计收入"：1000/10000/100000/1000000 金币）
  - 成就面板：网格展示，未解锁显示灰色+条件提示，已解锁显示彩色+解锁时间
  - 成就解锁时弹出全屏通知（成就图标 + 名称 + 动画效果）
  - 每个成就奖励不同：金币、钻石、特殊家具、称号
  - 隐藏成就：不显示解锁条件，直到解锁后才会展示
- **技术细节**：成就数据存储在 IndexedDB，每个成就有独立 ID。成就进度追踪复用 T7.1 的 `QuestTracker`

#### 任务 T7.4 — 随机事件系统

- **描述**：实现游戏中的随机事件触发、展示、执行流程
- **预估工时**：2 人天
- **优先级**：P1
- **前置依赖**：T7.1
- **验收标准**：
  - 实现 5 种随机事件（参照 PRD）：
    1. **美食博主探店**：等级≥5，随机触发 → 2 倍客流 3 游戏小时，所有顾客评价影响后续 3 天客流
    2. **顾客生日**：回头客来访 ≥3 次 → 弹出提示制作"生日特饮"，完成后获得 2💎
    3. **雨天**：随机天气触发 → 户外座位不可用，热饮需求翻倍，背景色调变冷
    4. **流浪猫来访**：完全随机 → 弹出收养选项，收养后可放置在店内（关联宠物系统）
    5. **咖啡大赛**：每月 1 次（游戏月） → 弹出参赛界面，提交最高品质咖啡，赢取稀有设备/家具
  - 事件触发条件由 `EventManager` 每游戏小时检查一次
  - 事件触发时弹出事件面板（模态框），包含事件描述、图标、选项按钮
  - 事件效果有时效性（如博主探店持续 3 游戏小时，结束后恢复正常）
  - 事件日志记录在 `EventLog` 中，可在"日志"面板查看
- **技术细节**：事件系统使用 `EventScheduler` 管理事件触发时机，每个事件有 `checkCondition()` 和 `execute()` 两个核心方法

#### 任务 T7.5 — 天气系统

- **描述**：实现天气系统（晴天/雨天），影响顾客行为和咖啡需求
- **预估工时**：1 人天
- **优先级**：P1
- **前置依赖**：T1.8、T7.4
- **验收标准**：

  - 每游戏天开始时随机决定当天天气，通过顶部状态栏图标显示
  - 雨天效果：

  - 天气切换平滑过渡（色调渐变 3 秒）
- **技术细节**：天气系统作为独立 `WeatherSystem`，在游戏主循环驱动。色调滤镜使用 PixiJS 的 `ColorMatrixFilter`

#### 任务 T7.6 — 任务面板 UI

#### 6.3.1 试玩节点

| 节点 | 时间 | 参与者 | 目标 |
|------|------|--------|------|
| **内部 Alpha** | W3 末 | 开发团队（3-5人） | 验证核心循环可玩性，发现明显 Bug |
| **内部 Beta** | W4 末（MVP 发布） | 开发团队 + 2-3 名同事 | 验证 MVP 完整性，收集初步体验反馈 |

#### 6.3.2 反馈收集方式

1. **试玩反馈表**（Google Form / 腾讯问卷）：
   - 总体满意度（1-5 分）
   - 最容易上手的部分 / 最难理解的部分
   - 游戏节奏感受（太快/太慢/合适）
   - 最想添加的功能（开放式问题）
   - Bug 反馈（附截图）
2. **埋点数据**（可选，使用自建轻量埋点）：
   - 各面板打开频率
   - 食谱制作次数分布
   - 平均游戏时长
   - 流失节点分析（最后操作是什么）
3. **直接观察**：
   - 试玩时安排 1 名开发者在旁观察
   - 记录玩家的困惑点、惊喜点、操作习惯

#### 6.3.3 反馈响应流程

```
试玩反馈收集 → 分类（Bug/建议/体验） → 优先级排序 → 分配 → 修复/实现 → 验证
```

### 7.2 各象限行动指南

#### 第一象限：优先做（Quick Win）— 高价值、低复杂度

**策略**：优先安排，快速完成以积累动能

- 这些任务通常是基础设施和配置类任务
- 1-2 人天可完成，交付确定性高
- 建议在迭代早期完成，为后续高复杂度任务打好基础

#### 第二象限：重点做（Strategic）— 高价值、高复杂度

**策略**：核心资源投入，充分设计后再动手

- 这是游戏的核心差异化竞争力所在
- 需要资深开发者主导，充分的技术设计评审
- 合理拆分 subtask，降低单次交付粒度
- 每个任务预留 20% 缓冲时间

#### 第三象限：按需做（Fill-in）— 低价值、低复杂度

**策略**：见缝插针，在等待或缓冲期完成

- 锦上添花的内容，提升游戏完整体验
- 适合初级开发者练手或作为"休息"任务
- 若时间紧张，V1.5 的此类任务可砍掉

#### 第四象限：谨慎做（Caution）— 低价值、高复杂度

**策略**：严格评估 ROI，大多数推迟到后续版本

- 节日活动复杂度高但价值有限（每年只用几周）
- 排行榜在无在线服务的情况下价值低
- 建议 MVP/V1.0 不做，V1.5 评估是否可砍

## 附录

### A. 工时估算说明

- **人天** = 1 名开发者全职工作 1 天（约 6 有效工时）
- 工时估算包含：设计（20%）+ 编码（50%）+ 自测（15%）+ 代码审查（10%）+ 文档更新（5%）
- 并行开发效率因子：3 人团队约 0.85（即 10 人天 / 0.85 = 约 11.8 日历人天），因为需要沟通和代码审查
- 每周按 5 个工作日计算

### B. 任务命名规范

任务编号格式：`T{周号}.{序号}`，如 T3.2 = 第 3 周第 2 个任务
